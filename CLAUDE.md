# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Overview

UMpack is a collection of 30 C++ libraries and tools for hypergraph partitioning, placement, and floorplanning. It contains approximately 445 source files (.cxx) and 283 header files (.h) organized as independent packages with well-defined dependencies. The codebase dates back to work at UCLA and University of Michigan (1995-2012).

## Build System

UMpack uses GNU Make as its primary build system, driven by Perl configuration scripts and shell wrappers.

### Initial Setup

```sh
perl config
```

This script:
- Detects your system and compiler (g++)
- Offers parallel compilation on multi-processor systems
- Creates `lib/` directory with symlinks to built libraries
- Prompts to build and test components
- Updates Makefiles with system-specific flags

### Building All Packages

```sh
perl config
# Follow the interactive prompts; decide explicitly whether to run tests
```

For the standard repository setup flow, there is a repo-local skill at
`skills/perl-config-regression/SKILL.md`.

When driving `perl config` manually in this repository:
- Taking default answers is fine through the `regression-only version` prompt.
- Do not stop on the first package compile failure. Let the package build
  phase finish so you get the full list of failing packages from one run.
- At `Would you like to test UMpack components now?`, answer yes only if the
  package build phase was clean. If some packages failed to build, answer no
  and skip tests unless running them is specifically needed to debug a known
  test problem.
- Do not auto-accept unexpected safety prompts about compiler/runtime bugs;
  stop and review them first.
- On the optimized default path, the default `memory-saving options` answer
  edits `ABKCommon/config.h` by removing the `SIGNAL_DIRECTIONS` block. That
  can change regression output, so restore `ABKCommon/config.h` before commit
  and rebuild affected packages after restoring it.

### Building Individual Packages

From a package directory (e.g., `HGraph/`):

```sh
./makeOpt-gnu          # Optimized build with -O3
./make-gnu             # Debug build with -DABKDEBUG -g
./makeOpt-gnu clean    # Remove compiled artifacts
```

The wrappers `make-gnu` and `makeOpt-gnu` are auto-generated by the config script. The actual build logic is in each package's `Makefile`, which defines:
- `THISLIB`: Package name (e.g., `HGraph`)
- `OBJS`: Object files to compile
- `LIBS`: Link dependencies (other UMpack libraries)
- Test executables (e.g., `HGraphTest0.exe`)

### Key Build Variables

- `LIBBASE`: Central library directory (created during config)
- `OPTFLAGS`: Optimization and debug flags
- `CCFLAGS`: Common compiler flags (includes `-fPIC` for shared libraries)
- `SODIRKEY`, `SONAMEKEY`: Shared library path and naming directives

Refer to `FIXES_AND_LESSONS.md` for build system lessons learned, especially regarding GNU toolchain consistency and library linking.

## Testing

### Running All Tests

After building with `perl config`, the script runs all package tests in dependency order. To rerun tests manually:

```sh
# From a package directory
./regression
```

### Running Package Tests

```sh
cd HGraph
./makeOpt-gnu test    # Builds test executables and runs ./regression
./regression          # Runs the regression script directly
```

### Understanding Regression Scripts

Each package has a `regression` executable shell script that:
1. Builds and runs test programs (e.g., `HGraphTest0.exe`, `HGraphTest1.exe`)
2. Generates output files (e.g., `new.out`)
3. Filters out volatile information (timing, platform, seed data)
4. Diffs filtered output against `expected.out`
5. Returns 0 only if diff is empty

Example from `HGraph/regression`:
```bash
./HGraphTest0.exe -f TESTS/addNetAreM.aux -saveNodes testOut
egrep -v "(emory|Created|User|Platform)" new.out > newout.notime
diff newout.notime expectedout.notime > diffs.notime
```

### Test Targets in Makefiles

Most packages define:
- `test`: Builds test executables and has the regression script available
- `lib`: Builds static (.a) and shared (.so) libraries

Example from `ABKCommon/Makefile`:
```makefile
TEST_EXES = ABKCommonTest0.exe ABKCommonTest1.exe ... ABKCommonTest22.exe
test : lib $(TEST_EXES)
```

### Single Test Execution

To run a single test executable directly:

```sh
cd HGraph
./HGraphTest0.exe -f TESTS/addNetAreM.aux
```

Test programs accept command-line parameters (see source .cxx files for details).

### Test Environment

Tests may require:
- `LD_LIBRARY_PATH` set if libraries are not found:
  ```sh
  export LD_LIBRARY_PATH="$PWD/lib:$LD_LIBRARY_PATH"
  ```
- `seeds.in` file for reproducible randomized tests (see "Random Seeds" below)

## Architecture and Packages

### Package Dependency Hierarchy

**Foundation (no dependencies on other UMpack libraries):**
- `ABKCommon`: Core utilities (timers, assertions, parameter parsing, I/O, random numbers)
- `Geoms`: Geometric primitives (points, intervals, bounding boxes)
- `Combi`: Combinatorial data structures
- `Ctainers`: Container utilities
- `Stats`: Statistics and random number generation (RNG)

**Mid-level (depend on foundation):**
- `HGraph`: Hypergraph representation (nodes, edges, nets) — widely used foundation
- `Placement`: Placement geometry and utilities
- `Constraints`: Constraint handling
- `Partitioning`: Partitioning problem formulation and I/O
- `PartEvals`, `PlaceEvals`: Evaluation metrics

**Higher-level (domain-specific algorithms):**
- `FMPart`, `MLPart`, `ShellPart`, `SmallPart`: Partitioning algorithms
- `RBPlace`, `SmallPlacement`, `SmallPlacers`: Placement algorithms
- `Capo`, `ParquetFP`, `ParquetDBFromRBP`: Floorplanning and refinement
- `HGraphWDims`, `ClusteredHGraph`, `FilledHier`, `GenHier`: Hierarchical structures
- `GeomTrees`, `CongestionMaps`: Geometry-based optimizations
- `Partitioners`, `PartLegality`, `AnalytPl`: Specialized partitioners and evaluators

Dependencies are enforced via `-l` flags in package Makefiles under the `LIBS` variable.

### Package Organization

Each package directory contains:
- `Makefile`: Build configuration (auto-updated by `perl config`)
- `*.h` and `*.cxx`: Source files
- `TESTS/` directory: Input test cases (.aux, .nets, etc.)
- `regression` script: Test driver
- `expected.out`: Golden output for regression tests
- `make-gnu`, `makeOpt-gnu`: Wrapper scripts (auto-generated)
- (Optional) README.md, DOCS/ folder

### Typical Package Structure

```
HGraph/
  ├── hgBase.h, hgBase.cxx          # Base hypergraph class
  ├── hgFixed.h, hgFixed.cxx        # Fixed-size hypergraph variant
  ├── hgRead.cxx, hgWrite.cxx       # File I/O
  ├── hgParam.h                     # Parameter handling
  ├── subHGraph.h, cluHGraph.h      # Specialized subgraph types
  ├── HGraphTest0.cxx ... Test3.cxx # Test programs
  ├── TESTS/                         # Test input files
  ├── expected.out                   # Golden output
  └── regression                     # Test script
```

## Header Includes and Dependencies

Include paths follow a pattern:

```cpp
#include "HGraph/hgFixed.h"        // Package with absolute path from repo root
#include "ABKCommon/abkassert.h"   // Focused foundation header
#include <cstring>                 // Standard headers for C/POSIX facilities
```

Header paths are set via `-I$(UCLA) -I.` in Makefiles, where `$(UCLA)` is the repository root. Always use relative includes from the repo root, not relative to the current package.

Do not add new direct includes of `ABKCommon/abkcommon.h` in source or public
headers. Prefer the narrow header that declares the facility you actually use,
for example:

- `ABKCommon/paramproc.h` for `StringParam`, `BoolParam`, `IntParam`, etc.
- `ABKCommon/abkio.h` for parsing manipulators such as `eathash()`,
  `needword()`, `needcaseword()`, and `skiptoeol`
- `ABKCommon/infolines.h` for `TimeStamp`, `User`, `Platform`, and `MemUsage`
- `ABKCommon/abktimer.h` for `Timer`
- `ABKCommon/abktypes.h` when a public header exposes `bit_vector`
- Standard headers such as `<cstring>`, `<strings.h>`, `<cmath>`, `<cfloat>`,
  and `<climits>` for C/POSIX library facilities and limits

## Random Seeds and Test Reproducibility

Several packages use deterministic seeding for reproducible randomized behavior:

- `seeds.in` (if present): First line is in-use flag (0 or 1), second line is seed value
- `seeds.out`: Runtime log of seeds consumed; exists after a run
- If no `seeds.in`: First random object picks a nondeterministic seed and logs it to `seeds.out`

To reproduce a previous run:
```sh
cp previousrun/seeds.out seeds.in
# Rerun the test with the same seed
```

The seed mechanism is implemented in `ABKCommon/seed.cxx` and `ABKCommon/abkseed.h`. Many regression scripts create or copy `seeds.in` before randomized tests.

## Compiler and Flags

**Compiler:** g++ (version 13+ typically)

**Key Flags:**
- `-fPIC`: Position-independent code (required for shared libraries)
- `-O3`: Optimization (makeOpt-gnu)
- `-g -DABKDEBUG`: Debug symbols (make-gnu)
- `-funroll-loops`: Loop unrolling in optimized builds
- `-Wall -pedantic`: Warnings (debug builds)

**Linking:**
- `-shared`: Build shared libraries
- `-Wl,-rpath,`: Set library search path at runtime
- `-Wl,-soname,`: Shared library name
- `-Wl,--start-group ... -Wl,--end-group`: Handle cyclic dependencies

If cyclic dependencies exist between libraries, they are wrapped in `--start-group`/`--end-group` to allow repeated symbol resolution.

## Common Development Patterns

### Parameter Parsing

ABKCommon provides `StringParam`, `IntParam`, and other parameter classes:

```cpp
#include "ABKCommon/paramproc.h"
StringParam filename("f", argc, argv);
if (filename.found()) { /* use filename.c_str() */ }
```

### Memory Usage Reporting

```cpp
#include "ABKCommon/infolines.h"
cout << MemUsage() << endl;  // Prints current memory stats
```

### Timing

```cpp
#include "ABKCommon/abktimer.h"
Timer timer;
timer.stop();
cout << "Time: " << timer << endl;
```

### String Use

Use standard library strings and streams for new code. The old `uofm::*`
allocator-backed containers have been removed from active C++ sources; do not
reintroduce them through compatibility guidance or stale examples.

## Notes for Development

- No compiler errors or warnings should appear (target of the build system)
- After changes to shared headers, perform a clean library rebuild (`./makeOpt-gnu clean && ./makeOpt-gnu`)
- After narrowing a public header, rebuild downstream package tests as well as
  the owning library; many historical failures came from tests depending on
  transitive parser or typedef exposure.
- Regression tests filter timing and platform-dependent output; changes must preserve algorithmic intent
- Test executables often require working directory to be the package directory (for TESTS/ access)
- If a test fails after build-system changes, try a clean library rebuild before debugging source code
